  <h2 className="mb-3 text-2xl md:text-3xl font-semibold tracking-tight">
    What an ERD Is and Why It Still Matters in 2025
  </h2>

  <div className="grid gap-6 md:grid-cols-2">
    <div className="rounded-2xl border bg-background p-5">
      <h3 className="mb-2 text-xl font-semibold tracking-tight">ERD basics in plain English</h3>
      <div className="text-sm text-muted-foreground">
        An ERD shows <strong>entities</strong> (things you store), <strong>attributes</strong> (fields),
        and <strong>relationships</strong> (how entities connect). You’ll also mark
        <strong> primary keys</strong> and <strong>foreign keys</strong> to keep data consistent.
      </div>
    </div>

    <div className="rounded-2xl border bg-background p-5">
      <h3 className="mb-2 text-xl font-semibold tracking-tight">Why teams still start with ERDs</h3>
      <div className="text-sm text-muted-foreground">
        ERDs create a shared language. Product owners see scope clearly, developers get structure
        early, and analysts understand the source of truth. This reduces defects and rework later.
      </div>
    </div>

  </div>

  <div className="mt-6 rounded-2xl border p-5">
    <h3 className="mb-2 text-xl font-semibold tracking-tight">Common pitfalls</h3>
    <ul className="list-disc pl-5 text-sm text-muted-foreground space-y-1">
      <li>Inconsistent names and data types</li>
      <li>Missing keys or unclear cardinalities</li>
      <li>Over/under-normalization that hurts performance or clarity</li>
    </ul>
  </div>
</section>

<section id="best-practices" className="not-prose my-12">
  <h2 className="mb-3 text-2xl md:text-3xl font-semibold tracking-tight">
    Best Practices for Clean, Maintainable ERDs
  </h2>

{" "}

<ul className="grid gap-3 md:grid-cols-2">
  <li className="rounded-xl border bg-background p-4">
    <strong>Model nouns and verbs first</strong>
    <div className="mt-1 text-sm text-muted-foreground">
      Entities as nouns (order, payment), relationships as verbs (order has{" "}
      <code>order_items</code>).
    </div>
  </li>
  <li className="rounded-xl border bg-background p-4">
    <strong>Enforce consistent naming</strong>
    <div className="mt-1 text-sm text-muted-foreground">
      Singular vs. plural, case, abbreviations.
    </div>
  </li>
  <li className="rounded-xl border bg-background p-4">
    <strong>Keys &amp; constraints</strong>
    <div className="mt-1 text-sm text-muted-foreground">
      Always define PKs, use FKs, add unique/check constraints.
    </div>
  </li>
  <li className="rounded-xl border bg-background p-4">
    <strong>Normalize, then pragmatically denormalize</strong>
    <div className="mt-1 text-sm text-muted-foreground">
      Clarity first; denormalize for clear wins.
    </div>
  </li>
  <li className="rounded-xl border bg-background p-4 md:col-span-2">
    <strong>Document decisions</strong>
    <div className="mt-1 text-sm text-muted-foreground">
      Capture assumptions, trade-offs, and term definitions near the diagram.
    </div>
  </li>
</ul>

  <div className="mt-6 rounded-2xl border p-5">
    <h3 className="mb-2 text-lg font-semibold">Limitations &amp; pro tips</h3>
    <ul className="list-disc pl-5 text-sm text-muted-foreground space-y-1">
      <li>Use manual edits for advanced constraints (partial indexes, composite uniqueness, partitioning).</li>
      <li>Prompt tip: include scope, entities, must-have attributes, and rules (e.g., “one user has many orders; email unique; payments pending/failed/succeeded”).</li>
    </ul>
  </div>
</section>
